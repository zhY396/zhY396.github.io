UML概述

  • 软件建模的UML类似于画电路图，作用是方便维护和修改设计
  • 管理组织OMG

软件开发过程

  • OOA：基本模型（对象、特征、关系）、主题（子系统）、交互图
  • OOD
  • 软件过程
    • 瀑布模型
    • 螺旋模型（原型开发）
    • RUP
      • 时间（动态结构）和核心工作流（静态结构）的二维坐标图
      • RUP的四种主要建模元素：角色（“谁”）、活动（“如何”）、制品（“某事”）、工作流（“何时” ）
      • RUP 阶段（phase）
        初始（先启）：定义整个项目的范围
        细化（精化）：制定项目计划、描述功能、建立架构框架
        构造（构建）：构造软件产品
        发布（产品化）：将软件产品移交到最终用户手中
      • 角色在工作流生成制品，通过活动达到目标
      • 9个核心工作流：6个核心过程工作流（Core Process Workflows）和3个核心支持工作流（Core Supporting Workflows）。
    • 敏捷开发：精益思维

用例图

  • 参与者（actor）也称为角色、活动者。
    为完成某个任务，是与系统进行交互的外部实体。
    相对系统而言，是外部实体所扮演的角色，不是虚拟的人

    按性质分类和按重要性分类

  • 用例是对系统、子系统或类与参与者交互动作序列的说明

  • 场景是用例的一次完整、具体的执行过程，用例是一组场景（脚本）的共同行为抽象

  • 事件流是场景中系统行为的一个特定序列

  • 参与者对用例的关系：关联，用实线表示

  • 用例之间的关系：

    • 泛化：空心三角形，实线
    • 包含：虚线箭头线，《include》构造型
    • 扩展：虚线箭头线，《extend》构造型，对基本用例的扩展，可以看作例外用例的实现，因为扩展用例依赖于基本用例的场景，所以不能独立存在

  • 用例描述：描述系统所做的事

类图

  • 消息

  • 接口（协议）

  • 服务

  • 可视性（Visibility）用以下可视性标记表示：

    • （公共），# （保护），– （私用），～（包）

  • 构造操作可以用构造型<>标示，也可以用下划线表示

  • 类（Class）的UML图标：属性的属主范围
    若属性的属主范围是实例，则该类的每一个实例对象都有一个自己的该属性的值
    若属性的属主范围是分类符，则对于该类本属性只有一个值，该类的每一个实例对象都持有此唯一的属性值，属性标注下划线

  • 例

  • 类操作：

    返回列表由逗号分隔的操作的返回值类型表达式组成，其格式为：返回类型 或 返回名字 = 类型，…

  • 类的UML图标：接口类、抽象类、模板类
    接口：一组操作的集合，只有操作的声明而没有实现
    标准图形、变体图形，没有属性
    不要求实现类和接口类概念本质上是一致的
    抽象类：不能被实例化的类，一般至少包含一个抽象操作
    类名、抽象操作名均为斜体
    模版类：一种参数化的类，在编译时把模版参数绑定到不同的数据类型，从而产生不同的类；模版化的类具有相同行为，但数据类型不同。

  • 分析类

    • 边界类
    • 实体类
    • 控制类

  • 对象

    • 对象图标不含有操作框。

  • 类之间的关系

  • 限定关联：订货：源类 作业线：目标类

  • 聚合关联：较弱的整体部分关系，空心菱形代表整体的一方

  • 组合关联：较强的整体部分关系，实心菱形代表整体的一方

  • 多态机制：

  • 类图模型建立过程
    寻找类：寻找备选类，从备选类中筛选出候选类。
    确定类关系，给关联加属性。
    确定类的职责：类所维护的信息、类提供的行为。

活动图

  • 特点：描述并发、没有循环控制机制、用状态作为基本的描述单元

  • 基本模型元素

    • 状态
      • 表示：圆角矩形
      • 动作状态：最小构造快，
      • 活动状态：动作状态的组合
    • 转移
      • 表示：实箭头线
      • 转移表达状态发生变化
    • 动作流：状态以及转移的组合
    • 分支
    • 分叉、汇合
      • 使用同步条（水平线）
      • 一个分叉和汇合构成一个线程
      • 一个线程加上监护条件变为条件线程，条件为假的时候可以不用等待该线程到达汇合
    • 泳道
      • 将活动状态分组表示，每一组代表某个对象执行该分组的活动，用垂直线划分泳道。

  • 动作流和分支可使用：[监护条件]/动作

  • 对象流

  • 活动分解

    • 简单活动
    • 组合活动

交互图

  • 行为视图：用例图、活动图、交互图、状态机图。
  • 顺序图标记符号
    对象类角色（Class Role）：矩形，符号与对象图相同。
    生命线（Lifeline）：虚线，表示对象生存期。
    消息（Message）：箭头线，表示对象间消息通信。
    激活（Activation）：矩形条，表示对象正在执行一些活动。
  • 对象类角色自左至右依次为参与者、边界、控制、实体等对象类型
  • 消息编号：消息号：消息
  • 消息类型
    • 简单消息
    • 返回消息
    • 同步消息
      • 接收者：被动对象
      • 表示：全箭头
      • 返回
    • 异步消息
      • 接收者：主动对象
      • 表示：半箭头
      • 回调
  • 组合片段
    • 不同区域用虚线分割，整个组合片段用矩形框表示
    • alt多条件 if xxx else yyy
    • opt单条件
    • loop
    • break

状态机图

  • 基本元素
    • 状态：圆角矩形
    • 转移：箭头线，箭头线上可以包含：事件[监护条件]/动作
    • 动作：
    • 活动
    • 事件
    • 判定：菱形
    • 分叉、汇合
  • 注意：并非所有可区分状态都有意义，如果区分状态后，没有针对每个状态有不同的处理方式，那就不必区分状态，也就是状态的粒度是可以变化的，取决于类方法的粒度。

构件图和部署图

构件图

  • 构件

    • 包含：构件名、构造型标注、类或对象

    • 构件两种依赖（依赖用虚线表示）

      • 开发时期依赖：构件图中使用
      • 调用依赖：部署图中使用

    • 构件之间相互依赖的构造型：

      • 《trace》，类的源代码依赖于类的设计
      • 《use》，某类使用了其他类
      • 《implements》
      • 《source》

    • 构件之间实箭头线表示使用

      一般在系统内部用实箭头线表示使用，外部用虚线连接系统的接口。

    • 构件可以放在包图中

    • 构件代表的是软件物理实现，例如源代码等

    • 构件可以区分型和值

  • 5个用于构件的标准构造型
    1．<>：说明一个构件可以在系统的节点上执行
    2．<>：说明一个构件是一个静态的或动态的对象库
    3．<

    >：说明一个构件代表的是一个数据库表
    4．<>： 说明一个构件代表的是一个文档，它包含的是源代码或数据
    5．<>：说明一个构件代表的是一个文档

组件图

  • 节点
    • 代表计算机物理资源，包括处理器或其他硬件（UML2.0，还可以代表执行环境，如J2EE容器、操作系统、数据库等）
    • 表示：斜视的立方体
    • 包含：对象名、构造型标注、构件或对象
    • 节点可以放在包图中
    • 节点可以区分型和值
  • 节点的关系
    • 通信关系（节点-节点）
      • 表示：实线、构造型标注
      • 表达节点之间的通信
    • 成为关系（构件-构件，对象-对象，构件-对象）
      • 表示：虚箭头线，《becomes》构造型、{time=}约束
      • 表达构件\对象不同时间点的变化

包图

  • 把语义上相近的、可能一起变更的模型元素组织在同一个包里，便于理解复杂的系统
  • 表示：包的图标是一个大矩形（内容框）的左上角带一个小矩形（名字框）
  • 在包名之前或之下，可用括在花括号中的文字（约束）说明包的性质，如“{abstract}”、“{version}”等
  • 包纯粹是一种概念性的建模工具
  • UML预定义的用于包的构造型有：<>、<>、<>、<>、<>等。
    构造型<>说明一个包仅仅是其他一些包的视图。
    构造型<>说明一个包代表模型架构。
    构造型<>说明一个包是另一个包的公共内容的服务代理。
    构造型<>说明一个包代表系统模型的一个独立部分，即子系统。
    构造型<>说明一个包代表系统模型。
    构造型<>和<>特别有助于管理大型系统模型。
  • 包之间的联系
    • 依赖
      • 表示：虚箭头线
      • 可以包含构造型，例如《import》、《access》、《merge》、《trace》
    • 泛化

约束

非形式化约束

  • 形式写出的约束应当用花括号“{ }”括起来
    放在所描述的模型元素的内部或紧靠所描述元素的地方
    也可放在注解的图标内，用虚线连接到所描述的模型元素
  • 标准约束：xor、subset

OCL对象约束语言

  • OCL 定义基本类型来描述数据：
    Boolean
    Integer
    Real

  • 格式：context name inv：

    ​ Boolean expression

  • name可以是：角色名、类名

  • 导航表达式：用于在Boolean expression中导航一个和name对应的类有关的类

    • name本身：self（注意：self是一个对象聚集）
    • name的相关对象:self.x.y （x和self有关联关系，y和x有关联关系）
    • 导航本身返回一个聚集
    • 限定关联的导航，对于限定符内的规定，用[]选取对应的划分
    • 可以使用关联类进行导航

  • 特性表达式：导航中角色的属性构成的集合

    • name本身的属性：self.a(a是self的属性，注意：self.a代表一个聚集)

  • 枚举

  • OCL定义三种类型的聚集:
    sets（集合）: 一个对象在集合中只能出现一次
    bags（袋）: 一个对象能够出现多次
    sequences（序列）: 对象按次序排列

  • 一个对象出发产生集合、一个集合出发产生袋

  • 聚集上的操作：

    • ->sum()

    • ->asSet()

    • ->size()

    • ->select()

    • ->collect() //通常不必使用，除非对属性进行了运算，返回一个袋

    • ->isEmpty()

    • ->includes()

    • ->includesAll()

    • ->forAll()

    • ->exists()

    • ps:

      • 括号里可以写布尔表达式，其中可以使用替换的名称，例如：

        “employees with a salary greater than 50,000”
        context Company inv:

                self.employee->select(**p:Person |**
        **p**.contract.grade.salary > 50000)

      • 操作返回聚集后可以继续导航

  • 操作符：and、or、not、xor、implies(蕴含)

  • 类不变量inv

  • 前置条件pre

  • 后置条件post，可以用name@pre表达前置条件时候的name

  • 对应三种构造型：<>、<>、<>

作业题目

  • 协作图中的消息：返回值:=方法 注意关联上方还有一个实心三角直线
  • 单向关联多重性只有一边有，如果有两个多重性是双向关联
  • 双向关联中可以用动词加上实心三角形，表示关联动作的方向
  • 对象图中的多重性问题、泛化问题
  • 可替换性原则：子类对象可以赋予父类对象
  • 画类图，首先要找到最大的公共操作，把公共操作的集合构造为一个基类，其他的类从基类里面扩展，公共操作里面可以有部分子类不需要或者需要限定的操作，在这些子类里面重写特定的操作即可。
  • 类中有其他类的成员，需要单向关联；如果是对象数组，多重性为*，如果是单个对象，为0…1

选择题

  • 建模不一定用标准图形符号
  • 建模不涉及底层实现细节
  • 成功建模：表示法、过程和工具
  • 创始人：Grady Booch、Ivar Jaconbson、James Rumbaugh
  • UML不统一各种软件过程
  • 用例视图描述外部
  • UML构成：图、视图、模型元素、通用机制
  • 通用机制：约束、构造型、标记值
  • simula67是第一个面向对象程序设计语言
  • RUP生命周期：初始、精化、构造、发布
  • 自关联有两个关联端，任何一个关联至少两个关联端
  • 活动指示要完成某项工作的动作或表示工作流的步骤
  • 顺序图标记符号：类元角色、生命线、激活、消息
  • 协作图和顺序图
  • 活动图着重表示活动到活动的控制流、状态机图着重表示模型元素可能的状态及其转移
  • 状态机图没有表现生命周期
  • 状态机图 全部状态属性
